1. Purpose + scope

This document specifies the HTTP API exposed by Odoo (module kebony_mes_api, to be built) to exchange production data with SIS MES via a middleware layer.

Three categories of data flow, matching the SIS spec (SIS_MES_Axapta_Summary v0.1):

FlowOdoo sideMES sideWho polls whom
White wood (raw material) stock stock.lot / stock.quant / stock.move ERP_RawMaterialPackage · ERP_Inventory Middleware polls Odoo
MO instructions mrp.production (+ kebony.dryer.load · kebony.autoclave.batch) ERP_ProductionOrder Middleware polls Odoo
Production feedback (stage events, consumption, quality) kebony.process.pack.event · stock.move · kebony.process.pack (mes_quality) MES_ProductionOrder · MES_UsedRawMaterialPackage · MES_ProducedPackage Middleware pushes to Odoo

Audience: SIS middleware team (Rinaldo + co.) designing the translation layer between the MES_ERP_Mailbox SQL schema and Odoo's HTTP API.

2. Architecture

┌──────────────┐ SQL schema ┌──────────────┐ HTTP / JSON ┌──────────────┐ │ │◄──────────────────┤ │◄──────────────────►│ │ │ MES │ MES_ERP_Mailbox │ Middleware │ /api/v1/mes/… │ Odoo │ │ (SIS) │ (shared DB) │ (SIS-built) │ │ kebony_mes_ │ │ │──────────────────►│ │ │ api │ └──────────────┘ └──────────────┘ └──────────────┘ emits on mailbox tables polls Odoo HTTP endpoints production + read/write every N seconds (this spec) events from ERP side + pushes events

Design principle: middleware isolates both sides. Odoo never talks to the MES database directly; MES never calls Odoo directly. Middleware owns translation, retry, buffering, cursor state, and scheduling.

Cadence recommendation (subject to SIS tuning):
  • Stock + inventory moves → poll every 60 s
  • MO instructions → poll every 60 s
  • Production events (stacking, autoclave, dryer, destack) → push immediately on MES event (or batched every 15–30 s if MES aggregates)

3. Out of scope

  • Middleware internals — language, runtime, deployment, scheduler, resilience. SIS owns the middleware; this spec only constrains the Odoo-facing contract.
  • MES database schema — already specified by SIS (SIS_MES_Axapta_Summary v0.1). This contract references MES fields only where helpful for mapping.
  • Inbound interactive flows (users creating MOs from MES UI) — Odoo remains the system of record for master data, procurement, invoicing, accounting. MES is production-execution only.
  • Finished-goods transactions back to ERP — handled by standard Odoo MO completion posting stock.move and valuation records. The API only covers what the current SIS spec lists.
  • Real-time streaming (WebSocket, SSE) — polling + synchronous POST is sufficient for the volumes at Kallo.

4. Conventions

TopicRule
Base URLhttps://<odoo-host>/api/v1/mes/
Version path prefix/v1/ — breaking changes bump to /v2/; backward-compatible additions stay on /v1/
FormatJSON in + JSON out; UTF-8; Content-Type: application/json
AuthenticationAPI key in header: X-API-Key: <key>. One key per middleware client. Rotatable via Odoo admin UI. No OAuth — single-client integration.
TimestampsISO 8601 UTC — e.g. 2026-04-23T14:12:05Z. No timezone offsets in payloads.
IDsIntegers are Odoo primary keys ("id": 4521). External identifiers (MES refs, supplier plates) are strings.
HTTP codes200 success · 201 created · 400 malformed payload · 401 bad API key · 404 unknown ID · 409 idempotency conflict · 422 validation failure · 500 server error
Rate limitingNot enforced at v1. Middleware should self-throttle to ≤ 60 rps total.

5. Cursor-based polling model

Every GET endpoint accepts a ?since=<cursor> query parameter and returns a new next_cursor in the response. Middleware stores the cursor and passes it on the next call.

Cursor is an opaque string (currently an ISO 8601 timestamp, but treated as opaque so we can evolve to (timestamp, id) tuples later).

Request:  GET /api/v1/mes/raw-packages?since=2026-04-23T10:00:00Z&limit=200
Response: 200 OK
{
  "data": [ /* up to 200 records */ ],
  "next_cursor": "2026-04-23T10:15:42Z",
  "has_more": true
}

If has_more = true, middleware should immediately re-poll with the new cursor. If has_more = false, wait for the next scheduled tick.

First call of a fresh middleware instance: omit since → server returns the most recent 200 records (for bootstrap) plus a cursor. Or pass since=0 for a full historical replay (if ever needed).

6. Endpoints — Odoo → middleware (pulled)

GET /api/v1/mes/raw-packages

Purpose: New/updated white wood packages available for MES to consume. Maps to SIS's ERP_RawMaterialPackage.

Query params: since (cursor), limit (default 200, max 1000)

{
  "data": [
    {
      "supplier_plate": "TS749697",       // stock.lot.name
      "article_number": "RAW-SP-100",     // product.default_code
      "article_name": "Scots Pine 1 inch, 3m",
      "start_amount": 120.0,              // boards at receipt
      "current_amount": 118.0,            // boards remaining
      "length_unit": 3.0,                 // x_studio_length_m (MES-mandatory)
      "blocked": false,                   // true = ClosedInAx / scrap / QC quarantine
      "location": "WH/KALLO/RAW",
      "received_datetime": "2026-04-20T09:15:00Z",
      "last_changed": "2026-04-23T08:30:12Z"
    }
  ],
  "next_cursor": "2026-04-23T08:30:12Z",
  "has_more": false
}

Mapping: supplier_plate → RawMaterialPackageIdent · blocked → Blocked = 1 when true.

GET /api/v1/mes/inventory-moves

Purpose: Wood-pack stock transactions since cursor. Maps to SIS's ERP_Inventory. Covers Sold, TransferFrom, TransferTo, Counting.

Query params: since, limit

{
  "data": [
    {
      "id": 88421,                         // stock.move.id
      "product_number": "RAW-SP-100",
      "package_ident": "TS749697",
      "amount": -2.0,                      // negative = out
      "transaction_type": "counting",      // "sold" | "transfer_from" | "transfer_to" | "counting" | "scrap"
      "location": "WH/KALLO/RAW",
      "location_dest": null,
      "timestamp": "2026-04-23T08:30:12Z",
      "invent_status": 0,                  // 1 = disable in MES
      "notes": null
    }
  ],
  "next_cursor": "2026-04-23T08:30:12Z",
  "has_more": false
}

GET /api/v1/mes/production-orders

Purpose: Active / planned MOs for MES to execute. Maps to SIS's ERP_ProductionOrder.

Query params: since, limit, state (optional filter: planned,confirmed,in_progress)

{
  "data": [
    {
      "production_order_ident": "MO/2026/00123",
      "dryer_load_ref": "DL-2026-0042",
      "dryer_id": "DRY2",                  // from kebony.equipment
      "autoclave_batches": [
        { "ref": "DL-2026-0042-AC01", "autoclave_id": "AUT1" },
        { "ref": "DL-2026-0042-AC02", "autoclave_id": "AUT1" }
      ],
      "product": {
        "article_number": "1931-I-10",
        "article_name": "Profile 1931, Imperial, 10 ft",
        "length_unit": 3.048
      },
      "planned_qty_boards": 480,
      "planned_m3": 1.23,
      "planned_start": "2026-04-24T06:00:00Z",
      "planned_finish": "2026-04-27T14:00:00Z",
      "operations": [10, 23, 26, 30],      // MES op codes
      "state": "planned"                    // "planned" | "confirmed" | "in_progress" | "done" | "cancelled"
    }
  ],
  "next_cursor": "2026-04-23T08:30:12Z",
  "has_more": false
}

7. Endpoints — middleware → Odoo (pushed)

POST /api/v1/mes/events

Purpose: Per-pack stage events emitted by MES. Maps to MES_ProducedPackage rows. One POST per batch of events (preferred) or per event. Writes to kebony.process.pack.event.

Request body:
{
  "events": [
    {
      "pack_number": 1234,                        // kebony.process.pack.pack_number
      "mes_produced_package_ident": "PP-019435-10-001",
      "mes_op_code": 10,                          // 10=stack · 23=autoclave · 26=dry · 30=destack
      "event_type": "stacked",                    // matches event_type selection
      "stage_before": null,
      "stage_after": "sa",
      "timestamp": "2026-04-24T06:12:45Z",
      "equipment_code": null,                     // "AUT1" / "DRY1" / etc. — optional
      "operator_code": "OP-12",                   // MES operator ID; mapped to res.users on Odoo side
      "mes_ref": "MES-evt-009912",
      "note": null
    }
  ]
}

Response: 201 Created
{
  "accepted": 1,
  "rejected": 0,
  "errors": []
}

Idempotency key: mes_produced_package_ident. Duplicate POSTs are safe — the server returns 200 OK with accepted: 0, already_present: 1.

POST /api/v1/mes/consumption

Purpose: Raw material consumption reported by MES. Maps to MES_UsedRawMaterialPackage. Supports whole loads, partial loads, and returns to raw storage. Writes stock.move records on Odoo side.

Request body:
{
  "transactions": [
    {
      "raw_package_ident": "TS720956",       // stock.lot.name
      "production_order_ident": "MO/2026/00123",
      "pack_number": 1234,                   // destination process pack
      "amount": 48.0,                        // boards consumed (positive)
      "direction": "consume",                // "consume" | "return_to_raw"
      "mes_op_code": 10,                     // operation that consumed it
      "timestamp": "2026-04-24T06:14:02Z",
      "mes_ref": "MES-cons-0045112"
    }
  ]
}

Response: 201 Created
{ "accepted": 1, "rejected": 0, "errors": [] }

Partial consumption handled natively. Middleware can emit "consume 48 of 60" followed by "return_to_raw 12" — Odoo creates two stock.move records. Idempotency key: mes_ref.

POST /api/v1/mes/produced-packages

Purpose: Per-step produced pack with quality + quantities. Maps to MES_ProducedPackage. Updates kebony.process.pack (actual quantities, mes_quality, stage advance) AND creates an event row.

Request body:
{
  "packages": [
    {
      "pack_number": 1234,
      "mes_produced_package_ident": "PP-019435-30-001",
      "mes_op_code": 30,                     // destacking
      "mes_quality": 1,                      // 1=prime · 2=secondary · 3=scrap
      "amount_boards": 47,                   // actual boards
      "amount_m3": 0.142,                    // optional, if MES computes
      "article_number": "1931-I-10",         // output SKU
      "timestamp": "2026-04-27T14:22:11Z",
      "mes_ref": "MES-prod-0009912"
    }
  ]
}

Response: 201 Created
{ "accepted": 1, "rejected": 0, "errors": [] }

Side effects on the Odoo side: pack stage advances (from aa to da on op 26, from da to fa on op 30), actual_lm / actual_m3 / actual_board_count are set, and a matching kebony.process.pack.event row is created.

8. Idempotency + dedup

EndpointIdempotency keyDuplicate behaviour
POST /eventsmes_produced_package_identSecond submit returns 200 OK + already_present: 1, no duplicate row
POST /consumptionmes_refSame
POST /produced-packagesmes_refSame — but if pack data differs, returns 409 Conflict with diff
GET endpointsn/aSafe by definition; same cursor returns same data

Middleware can replay failed POSTs without risk. Rule of thumb: on any 5xx or network error, retry with exponential backoff up to 5 times, then move to dead-letter queue.

9. Error handling + ingest queue

9.1 Error shape

Response: 422 Unprocessable Entity
{
  "accepted": 2,
  "rejected": 1,
  "errors": [
    {
      "index": 2,                         // index in the submitted array
      "error_code": "pack_not_found",
      "error_message": "No kebony.process.pack with pack_number=9999 found.",
      "payload": { /* the rejected item, verbatim */ }
    }
  ]
}

Partial success is allowed — accepted items are committed, rejected items are returned in errors for middleware retry or manual handling.

9.2 Ingest queue on Odoo side

Beyond the synchronous response, Odoo keeps a failed-ingest log (kebony.mes.ingest.queue) with:

  • Raw payload (JSON)
  • Error code + message
  • Timestamp received + last retry
  • Retry count
  • Status: pending / resolved / dead

QA / IT ops have a view to inspect + replay these. SIS middleware can also call POST /api/v1/mes/events again with the original payload; idempotency prevents double-writes.

10. Open questions for SIS

Need answers before implementation starts
  1. Quality 1 vs 2 — SIS doc explicitly defines Quality = 3 (scrap) but not 1 / 2. Confirm meaning (prime / secondary?) so we can label the selection on Odoo side.
  2. Operator mapping — does MES emit stable operator codes on MES_ProducedPackage, or just a name? We need a stable key to map to res.users.
  3. Equipment identifier — MES reports which physical dryer (1/2/3)? If yes, under what field? Critical for capacity tracking on Odoo side.
  4. Consumption timing — when the stacking machine loads 60 boards and later returns 12, does MES emit two MES_UsedRawMaterialPackage rows (one consume +60, one return -12), or one net row (+48)? Affects how we model direction.
  5. MO cancellation flow — if Odoo cancels a planned MO, how does that propagate to MES? Do we need DELETE /api/v1/mes/production-orders/{ident} or a status change via another endpoint?
  6. Polling or push for events — is middleware happy polling MES's output tables every 15–30 s, or should MES push? (Affects cadence of the inbound POST /events.)
  7. Article master sync — this spec covers transactions. Master data (articles, planning family, equipment) is also needed in MES. Does middleware pull that separately, or should we add endpoints here?
  8. Authentication ownership — who issues and rotates the API key? Proposed: Odoo admin UI, rotated quarterly; SIS stores in middleware config.
Already agreed
  • Middleware owns all translation + retry
  • Odoo exposes HTTP/JSON — no direct DB access
  • Cursor-based polling, not real-time streaming
  • Idempotency via stable MES identifiers
  • Partial success on batch POSTs
  • Error queue on Odoo side for replay
