1. Purpose & scope

This guide describes the HTTP API exposed by Odoo 19 (module kebony_manufacturing) for the SIS MES at the BNV Antwerp plant. It implements the contract defined in mes_api_contract.html (v0.1) and is the operational reference for the team building the middleware between the MES and the ERP.

Two directions of exchange, all under /api/v1/mes/:

DirectionTransportEndpointsPurpose
ERP → MES GET /articles · /article-classes · /raw-packages · /production-orders Middleware pulls product master, classifications, raw stock, planned production
MES → ERP POST /produced-package · /production-order/actuals · /raw-material-consumption Middleware pushes operations, quality, MO actuals, raw consumption
BNV-only. All exchanges are scoped to the Kebony BNV company (Antwerp plant). US, Norway and Luxembourg entities are out of scope and explicitly filtered out by the controller (allowed_company_ids = [BNV]).

Network requirements:

  • Outbound from middleware → Odoo on port 443
  • No inbound from Odoo to middleware (pull-based for ERP→MES, push-based for MES→ERP)

3. Authentication & key lifecycle

3.1 Mechanism

API key in either header (server accepts both):

X-API-Key: <key>
Authorization: Bearer <key>

The key is opaque (40-char hex), tied to a single Odoo technical user mes_middleware (display name "MES Middleware"). All actions are audited under this user. The user has no interactive password by default and cannot log in to the Odoo web UI — keys are the only credential.

3.2 Key generation (middleware self-service)

The SIS middleware operator generates their own API key by logging in as the mes_middleware user. The user is pre-provisioned with an initial password — change it on first login.

FieldValue
Loginmes_middleware
Initial passwordMES12345 Change on first login
Login URL<env-base-url>/web/login
  1. Open <base-url>/web/login in a browser.
  2. Login: mes_middleware · Password: MES12345.
  3. Top-right avatar → My Profile (or Preferences).
  4. Tab Account Security → button New API Key (or Add API Key).
  5. Description: MES middleware – <env> – YYYY-MM-DD. Duration: per security policy (recommended 1 year max).
  6. Click Generate Key (Odoo will re-prompt for the current password as a confirmation step).
  7. Copy the key immediately — Odoo only stores a hash; the value is shown once.
  8. Store the key in the middleware's secret store (vault / env var / 1Password). Never commit to source.
  9. Click Change Password (same screen) to replace MES12345 with a strong, randomly-generated password held by the middleware operator.
Change MES12345 immediately after first login. The initial password is shared knowledge by design (so SIS can self-serve the first key generation), but must be replaced before production. The API key — not the password — is the long-term credential.

3.3 Rotation & revocation

  • Rotate: generate new key → deploy to middleware → verify health → revoke old key. Keep ≥ 5 minutes overlap.
  • Revoke: delete the line on the same screen. Stops working on the next request (no cache).

3.4 Health probe

/api/v1/mes/ping is unauthenticated and verifies routing & TLS without exposing any data:

curl https://kebonyprod-test-31051334.dev.odoo.com/api/v1/mes/ping
# {"status": "ok", "service": "kebony-mes-api", "version": "1.0.0"}

Use for liveness checks. Do not use to verify key validity — call /articles and check 200 vs 401 instead.

4. Conventions

4.1 Content types

  • Request bodies (POST): application/json, UTF-8.
  • Responses: application/json, UTF-8.
  • GET query params: standard URL-encoded.

4.2 Datetimes

All datetimes are ISO 8601 with explicit timezone — Z (UTC) preferred:

FormatStatus
2026-04-29T08:15:00ZAccepted
2026-04-29T10:15:00+02:00Accepted
2026-04-29 08:15:00Rejected — no timezone

4.3 Identifiers

FieldFormatOwnerExample
ArticleNum<part>:<length_dm>::: (SIS canonical)ERP (transformed from default_code / name)1003:35:::
KebonyDefaultCode<part>-M-<length_m>ERP (informational mirror of native Kebony SKU)1003-M-3.5
ArticleClassIDstring codeERP (kebony.planning.family.code)KSP1C
ProductionOrderIdent{warehouse}/MO/NNNNNERP (mrp.production.name)Kallo/MO/00005

PackRef (legacy){STAGE}-{NNNNN}ERP (kebony.process.pack.name)SA-00123
Pack identity invariant — CRITICAL. One RawMaterialPackageIdent = one physical pack = boards all identical. A pack is never physically split while keeping the same identifier — if material is removed, the original pack KEEPS its number (with reduced quantity); the removed portion gets a NEW identifier. Duplicate RawMaterialPackageIdent in a /raw-packages response is a data integrity error, not a normal split.

Article number transformation: Kebony stores SKUs as <part>-M-<length_m> (metric only — imperial -I- codes are excluded from the MES feed). The wire format encodes length in decimeters:

Kebony nativeSIS canonical (ArticleNum)Notes
1003-M-3.51003:35:::3.5 m × 10 = 35 dm
1127-M-3.51127:35:::same length, different part
2489-M-4.82489:48:::—
1003-I-12excludedImperial — not on MES feed

Three trailing colons match the SIS layout (<code>:<length>:<empty>:<empty>:). The two empty trailing fields are reserved for SIS-side variant / grade attributes Kebony doesn't currently populate.

Defect reason codes live inside MES only and are aggregated to Quality 1/2/3 before reaching ERP. The full reason catalogue (when shared) is loaded as ERP-side seed data, not interface payload.

4.6 Errors

All non-2xx responses share the same JSON shape:

{"error": "<human-readable description>"}

5. Endpoint reference

Detailed JSON schemas (types, enums, required vs optional) live in the OpenAPI spec at addons/kebony_manufacturing/static/openapi/mes_api.json. The summaries below give the practical day-to-day picture.

GET /api/v1/mes/ping

Health probe. No auth.

Response:

{"status": "ok", "service": "kebony-mes-api", "version": "1.1.0"}

GET /api/v1/mes/articles

Article master + on-hand stock totals. Scope: white-wood SKUs only (x_studio_product_classification = "wood", metric codes only). Imperial -I- codes are excluded. BNV only.

Query params (all optional):

  • updated_since — ISO 8601; only return articles with WriteDate >= this
  • planning_family — filter by planning family code (e.g. KSP1C)

Response (live capture from kebonyprod-test-31051334):

{
  "count": 1329,
  "articles": [
    {
      "ArticleNum": "1127:35:::",
      "ArticleName": "1127-M-3.5",
      "ArticleClassID": "KSP1C",
      "LengthUnit": 3.5,
      "BoardsPerPack": 95,
      "VolumePerMeter_m3": 0.00304,
      "PlanningFamily": "KSP1C",
      "UoM": "LM",
      "QtyAvailable": 2735.0,
      "ClosedInAx": false,
      "WriteDate": "2026-04-30T06:55:19",
      "KebonyDefaultCode": "1127-M-3.5"
    }
  ]
}

Field semantics:

  • ArticleNum — SIS canonical format <part>:<length_dm>:::. Use as the primary key on the MES side.
  • KebonyDefaultCode — Kebony's native code (mirror of default_code / name). Cross-reference / debugging only.
  • ArticleClassID — joins to /article-classes; equals the article's planning family code.
  • LengthUnit — board length in metres. Mandatory for MES running-meter computation. Maps from x_studio_length_m.
  • VolumePerMeter_m3 — m³ per linear metre. Kebony doctrine: total_m3 = total_lm × VolumePerMeter_m3.
  • ClosedInAx — true when product is deactivated. MES drops these from working set.
  • QtyAvailable — total internal-location on-hand for this article (BNV only).

Polling cadence: 5–15 min with updated_since. Full pulls on cold-start only.

GET /api/v1/mes/article-classes

Article classification master (= SIS ERP_ArticleClass). Returns the planning families used as the canonical class for each white-wood article.

Query params:

  • code — exact ArticleClassID match

Response (live capture):

{

  "article_classes": [
    {"ArticleClassID": "KSP1C",   "ArticleClassName": "KSP 1\" Cladding"},
    {"ArticleClassID": "KSP15T",  "ArticleClassName": "KSP 1.5\" Terrace"},

    {"ArticleClassID": "KSP2CON", "ArticleClassName": "KSP 2\" Construction"},
    {"ArticleClassID": "KRRS1",   "ArticleClassName": "KRRS 1\""}

}

Polling cadence: cache aggressively — this list changes a few times per year. Refresh nightly.

GET /api/v1/mes/raw-packages

Per-lot inventory with location and SIS-spec flags. Same white-wood scope as /articles.

Query params:

  • article_num — accepts both Kebony native (1127-M-3.5) and SIS format (1127:35:::); SIS-format filters by part-prefix, pass Kebony format for an exact match
  • location — substring match on full location path

Response (live capture):

{
  "count": 5,
  "packages": [
    {
      "RawMaterialPackageIdent": "0000070",
      "ArticleNum": "1127:35:::",
      "Amount": 467.0,
      "ReservedQty": 0.0,
      "UoM": "LM",
      "Location": "Kallo/Stock",
      "InventStatus": 1,
      "Blocked": false,
      "LotRef": 2062,
      "KebonyDefaultCode": "1127-M-3.5"
    }
  ]
}
  • RawMaterialPackageIdent — supplier plate / lot name. Unique per response per the pack identity invariant (§4.3). Duplicates are anomalies.
  • ArticleNum — SIS canonical (matches /articles).
  • InventStatus — 1 while Amount > 0, else 0 (deactivation flag).
  • Blocked — currently always false; sourced from stock.lot.x_kebony_blocked once that field exists.

Polling cadence: 1–5 min depending on MES throughput sensitivity.

GET /api/v1/mes/production-orders

Planned production (= SIS ERP_ProductionOrder). MO-centric — internal Kebony dryer / autoclave / pack planning is no longer exposed.

Query params:

  • state — draft / confirmed / progress (default = draft + confirmed + progress)
  • ref — exact ProductionOrderIdent lookup
  • article_num — accepts both Kebony native and SIS format

Response shape (live capture):

{
  "count": 1,
  "production_orders": [
    {
      "ProductionOrderIdent": "Kallo/MO/00005",
      "Id": 62,
      "Active": true,
      "ArticleNum": "1127:35:::",
      "KebonyDefaultCode": "1127-M-3.5",
      "PlannedQty": 567.0,
      "UoM": "LM",
      "PlannedStart": "2026-04-29T06:00:00",
      "PlannedFinish": "2026-05-01T18:00:00",
      "State": "confirmed",
      "OperationCodes": [10, 23, 26, 30],
      "OperationStatus": {"10": "planned", "23": "planned", "26": "planned", "30": "planned"},
      "KebonyDryerLoadRef": "DL-2026-0042"
    }
  ]
}
  • OperationCodes — fixed [10, 23, 26, 30] for Kebony wood production (Stacking / Autoclave / Drying / De-stacking).
  • OperationStatus — best-effort coarse mapping from MO state. Will be refined to per-operation granularity in v1.2.
  • KebonyDryerLoadRef — pure cross-reference for support / debugging. MES should ignore it.
  • White-wood scope only: chemical, accessory, and finished-goods MOs are excluded.

Polling cadence: 1–3 min.

POST /api/v1/mes/produced-package

Called once per operation step per pack. Optional Quality + actuals fields update the parent process pack in the same request — typically only on Op 30.

Request body (preferred — by ProductionOrderIdent):

{
  "OperationCode": 10,
  "ProducedPackageIdent": "MES-SA-00123-1",
  "ProductionOrderIdent": "Kallo/MO/00005",
  "Timestamp": "2026-04-29T08:15:00Z",
  "MesRef": "WO-44912"
}

Op 30 example with Quality + actuals:

{
  "OperationCode": 30,
  "ProducedPackageIdent": "MES-FA-00123-4",
  "ProductionOrderIdent": "Kallo/MO/00005",
  "Quality": 1,
  "ActualLM": 565.5,
  "ActualM3": 1.90,
  "ActualBoardCount": 161,
  "OutputPackQtyActual": 0.99,
  "Timestamp": "2026-05-01T18:42:00Z"
}

Required: OperationCode, ProducedPackageIdent, AND either ProductionOrderIdent (preferred per SIS spec) or PackRef (legacy Kebony-side). All others optional.

When ProductionOrderIdent is supplied, the API auto-resolves the right kebony.process.pack on that MO that hasn't yet received an event of this type. Falling back to PackRef is supported for backward compatibility but ProductionOrderIdent is the SIS-canonical key.

Response:

{
  "status": "ok",
  "event_id": 17234,
  "pack_ref": "FA-00123",
  "event_type": "destacked",
  "updated_pack_fields": ["mes_quality", "actual_lm", "actual_m3", "actual_board_count", "output_pack_qty_actual"]
}
Idempotency: the API does not deduplicate. Retries create duplicate events. Middleware tracks delivery state by ProducedPackageIdent and only re-POSTs on confirmed failures (5xx / network).

POST /api/v1/mes/production-order/actuals

MES informs ERP that an MO has started (or finished). Maps to SIS MES_ProductionOrder.ActualStartTime.

Request body:

{
  "ProductionOrderIdent": "Kallo/MO/00005",
  "ActualStartTime": "2026-04-29T06:00:00Z",
  "ActualFinishTime": "2026-05-01T18:30:00Z"
}

Required: ProductionOrderIdent. Times optional — send only what you have. Per spec, ActualStartTime is the primary signal.

Response:

{
  "status": "ok",
  "production_order": "Kallo/MO/00005",
  "updated_fields": ["date_start", "date_finished"]
}

This endpoint is idempotent on the ERP side (write, not append) — safe to retry.

POST /api/v1/mes/raw-material-consumption

Audit log of MES loading raw packs into the stacking machine, plus partial returns to raw storage. Maps to SIS MES_UsedRawMaterialPackage. Quality is NOT used here — raw consumption is uncategorised; quality only applies to produced packs.

Request body:

{
  "ProductionOrderIdent": "Kallo/MO/00005",
  "RawMaterialPackageIdent": "0000070",
  "OpKind": "load_partial",
  "Amount": 320.5,
  "Timestamp": "2026-04-29T08:15:00Z",
  "MesRef": "WO-44912",
  "Note": "Loaded 320.5 LM, returned 147 LM remainder."
}

Required: ProductionOrderIdent, RawMaterialPackageIdent, OpKind, Amount.

OpKind values:

ValueMeaning
load_fullWhole raw pack loaded into machine
load_partialPart of pack loaded; remainder still in storage
return_fullWhole pack returned to raw storage (e.g. cancelled)
return_partialPartial remainder returned (the "spare boards" SIS doc references)

Response:

{
  "status": "ok",
  "event_id": 8421,
  "production_order": "Kallo/MO/00005",
  "raw_pack": "0000070",
  "op_kind": "load_partial"
}
Behaviour: v1.1 ships the recording surface only — the event lands in kebony.mes.raw.consumption.event for audit. It does not drive Odoo stock moves automatically. Reconciliation against stock.quant is a separate process.
Idempotency: the API does NOT deduplicate. Track delivery state middleware-side keyed by your own correlation ID.

6. End-to-end happy path

ERP creates Kallo/MO/00005 (state=confirmed, article 1127:35:::, 567 LM) │ ▼ middleware polls /production-orders MES picks up Kallo/MO/00005, allocates raw packs, drives the line │ ▼ Raw pack 0000070 loaded into stacker (partial) POST /raw-material-consumption {ProductionOrderIdent: "Kallo/MO/00005", RawMaterialPackageIdent: "0000070", OpKind: "load_partial", Amount: 320.5} │ ▼ MO actually starts POST /production-order/actuals {ProductionOrderIdent: "Kallo/MO/00005", ActualStartTime: "2026-04-29T06:00:00Z"} │ ▼ Op 10 — Stacking (per pack) POST /produced-package {OperationCode: 10, ProductionOrderIdent: "Kallo/MO/00005", ProducedPackageIdent: "MES-SA-00123-1", ...} │ ▼ Op 23 — Autoclave charging POST /produced-package {OperationCode: 23, ProductionOrderIdent: "Kallo/MO/00005", ProducedPackageIdent: "MES-AA-00123-1", ...} │ ▼ Op 26 — Drying complete POST /produced-package {OperationCode: 26, ProductionOrderIdent: "Kallo/MO/00005", ProducedPackageIdent: "MES-DA-00123-1", ...} │ ▼ Op 30 — De-stacking with Quality + actuals POST /produced-package {OperationCode: 30, ProductionOrderIdent: "Kallo/MO/00005", ProducedPackageIdent: "MES-FA-00123-1", Quality: 1, ActualLM: 565.5, ActualM3: 1.9, ...} │ ▼ Spare boards returned to raw storage (gets a NEW lot identifier) POST /raw-material-consumption {ProductionOrderIdent: "Kallo/MO/00005", RawMaterialPackageIdent: "0000070-spare", OpKind: "return_partial", Amount: 147.0} │ ▼ MO finishes POST /production-order/actuals {ProductionOrderIdent: "Kallo/MO/00005", ActualFinishTime: "2026-05-01T18:30:00Z"}
Stage prefix in PackRef. Legacy PackRef path only — when using ProductionOrderIdent (preferred) the API auto-resolves the right pack and prefix is moot. If MES still sends PackRef, it can either use the operation-appropriate prefix (SA- / AA- / DA- / FA-) or always send SA- and rely on ERP resolution. VVF recommends operation-appropriate for clarity in event audit. Confirm with VVF before implementation.

7. Operational guidance

7.1 Retry policy

StatusRetry?Strategy
5xxYesExponential backoff: 1s · 2s · 5s · 10s · 30s, then alert
4xxNoDeterministic failure. Log, alert, manual triage
Network / timeoutYesSame backoff as 5xx

7.2 Logging requirements

For every call (in & out), middleware logs:

  • Direction, endpoint, HTTP status, latency
  • Correlation ID (your own — e.g. MesRef)
  • For POSTs: payload (redact API key)
  • For GETs: count returned (not full body)

Retain ≥ 90 days for production troubleshooting.

7.3 Idempotency strategy

  • produced-package is not deduplicated server-side. Persist ProducedPackageIdent + ERP event_id in middleware state on success. On retry, check first; only re-POST if not previously confirmed.
  • raw-material-consumption is not deduplicated server-side either. Track delivery state by your own correlation ID (MesRef) and only re-POST on confirmed failures.
  • production-order/actuals is idempotent (write, not append) — safe to retry.

7.4 Backpressure

ERP can absorb roughly N requests/sec/instance (TBD load-tested). For burst protection: cap concurrent in-flight requests (recommended 4–8). Queue and drain rather than parallelising aggressively.

7.5 Time skew

Both systems run NTP. Skew > 60 s causes Timestamp ordering anomalies in event histories.

8. Quick-start with curl

export BASE="https://kebonyprod-test-31051334.dev.odoo.com"
export KEY="<your-api-key>"

# Health
curl "$BASE/api/v1/mes/ping"
# → {"status":"ok","service":"kebony-mes-api","version":"1.1.0"}

# Articles (white-wood, SIS code format)
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/articles?planning_family=KSP1C" | jq .count

# Article classes (planning families = ERP_ArticleClass)
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/article-classes"

# Raw packages
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/raw-packages" | jq '.packages | length'

# Planned production orders (replaces dryer-loads)
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/production-orders?state=confirmed"

# Report stacking event (preferred — by ProductionOrderIdent)
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"OperationCode":10,"ProducedPackageIdent":"MES-SA-00123-1","ProductionOrderIdent":"Kallo/MO/00005","Timestamp":"2026-04-29T08:15:00Z"}' \
  "$BASE/api/v1/mes/produced-package"

# Report MO start time
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"ProductionOrderIdent":"Kallo/MO/00005","ActualStartTime":"2026-04-29T06:00:00Z"}' \
  "$BASE/api/v1/mes/production-order/actuals"

# Report raw consumption (partial load)
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"ProductionOrderIdent":"Kallo/MO/00005","RawMaterialPackageIdent":"0000070","OpKind":"load_partial","Amount":320.5,"Timestamp":"2026-04-29T08:15:00Z"}' \
  "$BASE/api/v1/mes/raw-material-consumption"
