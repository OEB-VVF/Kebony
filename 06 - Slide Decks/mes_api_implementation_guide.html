1. Purpose & scope

This guide describes the HTTP API exposed by Odoo 19 (module kebony_manufacturing) for the SIS MES at the BNV Antwerp plant. It implements the contract defined in mes_api_contract.html (v0.1) and is the operational reference for the team building the middleware between the MES and the ERP.

Two directions of exchange, all under /api/v1/mes/:

DirectionTransportEndpointsPurpose
ERP → MES GET /articles · /raw-packages · /dryer-loads Middleware pulls product master, raw stock, planned production
MES → ERP POST /produced-package · /dryer-load/actuals Middleware pushes operations, quality, actuals
BNV-only. All exchanges are scoped to the Kebony BNV company (Antwerp plant). US, Norway and Luxembourg entities are out of scope and explicitly filtered out by the controller (allowed_company_ids = [BNV]).

Network requirements:

  • HTTPS only (TLS 1.2+)
  • Outbound from middleware → Odoo on port 443
  • No inbound from Odoo to middleware (pull-based for ERP→MES, push-based for MES→ERP)

3. Authentication & key lifecycle

3.1 Mechanism

API key in either header (server accepts both):

X-API-Key: <key>
Authorization: Bearer <key>

The key is opaque (40-char hex), tied to a single Odoo technical user mes_middleware (display name "MES Middleware"). All actions are audited under this user. The user has no interactive password by default and cannot log in to the Odoo web UI — keys are the only credential.

3.2 Key generation (middleware self-service)

The SIS middleware operator generates their own API key by logging in as the mes_middleware user. The user is pre-provisioned with an initial password — change it on first login.

FieldValue
Loginmes_middleware
Initial passwordMES12345 Change on first login
Login URL<env-base-url>/web/login
  1. Open <base-url>/web/login in a browser.
  2. Login: mes_middleware · Password: MES12345.
  3. Top-right avatar → My Profile (or Preferences).
  4. Tab Account Security → button New API Key (or Add API Key).
  5. Description: MES middleware – <env> – YYYY-MM-DD. Duration: per security policy (recommended 1 year max).
  6. Click Generate Key (Odoo will re-prompt for the current password as a confirmation step).
  7. Copy the key immediately — Odoo only stores a hash; the value is shown once.
  8. Store the key in the middleware's secret store (vault / env var / 1Password). Never commit to source.
  9. Click Change Password (same screen) to replace MES12345 with a strong, randomly-generated password held by the middleware operator.
Change MES12345 immediately after first login. The initial password is shared knowledge by design (so SIS can self-serve the first key generation), but must be replaced before production. The API key — not the password — is the long-term credential.

3.3 Rotation & revocation

  • Rotate: generate new key → deploy to middleware → verify health → revoke old key. Keep ≥ 5 minutes overlap.
  • Revoke: delete the line on the same screen. Stops working on the next request (no cache).

3.4 Health probe

/api/v1/mes/ping is unauthenticated and verifies routing & TLS without exposing any data:

curl https://kebonyprod-test-31051334.dev.odoo.com/api/v1/mes/ping
# {"status": "ok", "service": "kebony-mes-api", "version": "1.0.0"}

Use for liveness checks. Do not use to verify key validity — call /articles and check 200 vs 401 instead.

4. Conventions

4.1 Content types

  • Request bodies (POST): application/json, UTF-8.
  • Responses: application/json, UTF-8.
  • GET query params: standard URL-encoded.

4.2 Datetimes

All datetimes are ISO 8601 with explicit timezone — Z (UTC) preferred:

FormatStatus
2026-04-29T08:15:00ZAccepted
2026-04-29T10:15:00+02:00Accepted
2026-04-29 08:15:00Rejected — no timezone

4.5 Quality codes

CodeMeaningStatus

Defect reason codes live inside MES only and are aggregated to Quality 1/2/3 before reaching ERP. The full reason catalogue (when shared) is loaded as ERP-side seed data, not interface payload.

4.6 Errors

All non-2xx responses share the same JSON shape:

{"error": "<human-readable description>"}
StatusWhenMiddleware action
200Success—
400Validation (missing required, bad enum, malformed datetime)Do not retry · log + alert · manual triage
401Missing or invalid API keyCheck key rotation; do not retry blindly
404PackRef / DryerLoadRef not foundPossible race; retry once after 30 s, then alert
500Server-side failureExponential backoff; alert on persistence

5. Endpoint reference

Detailed JSON schemas (types, enums, required vs optional) live in the OpenAPI spec at addons/kebony_manufacturing/static/openapi/mes_api.json. The summaries below give the practical day-to-day picture.

GET /api/v1/mes/ping

Health probe. No auth.

Response:

{"status": "ok", "service": "kebony-mes-api", "version": "1.0.0"}

GET /api/v1/mes/articles

Article master + on-hand stock totals (BNV only).

Query params (all optional):

  • updated_since — ISO 8601; only return articles with WriteDate >= this
  • planning_family — filter by planning family code (e.g. KRRS1)

Response (excerpt):

{
  "count": 412,
  "articles": [
    {
      "ArticleNum": "2468",
      "ArticleName": "1931-I-10",
      "LengthUnit": 3.05,
      "BoardsPerPack": 95,
      "VolumePerMeter_m3": 0.005,
      "PlanningFamily": "KRRS1",
      "UoM": "LF",
      "QtyAvailable": 2850.0,
      "ClosedInAx": false,
      "WriteDate": "2026-04-28T06:55:19"
    }
  ]
}

Field semantics:

  • LengthUnit — board length in metres. Mandatory for MES running-meter computation. Maps from x_studio_length_m.
  • VolumePerMeter_m3 — m³ per linear metre. Kebony doctrine: total_m3 = total_lm × VolumePerMeter_m3.
  • ClosedInAx — true when product is deactivated. MES drops these from working set.
  • QtyAvailable — total internal-location on-hand for this article (BNV only).

Polling cadence: 5–15 min with updated_since. Full pulls on cold-start only.

GET /api/v1/mes/raw-packages

Per-lot inventory with location and SIS-spec flags.

Query params:

  • article_num — filter by ArticleNum
  • location — substring match on full location path

Response (excerpt):

{
  "count": 47,
  "packages": [
    {
      "RawMaterialPackageIdent": "TS749697",
      "ArticleNum": "WW-RAD-95x32-3000",
      "Amount": 1928.67,
      "ReservedQty": 0.0,
      "UoM": "LF",
      "Location": "Kallo/Stock/RawMaterial",
      "InventStatus": 1,
      "Blocked": false,
      "LotRef": 1158
    }
  ]
}
  • RawMaterialPackageIdent — supplier plate / lot name. Stable across the pack's lifetime.
  • InventStatus — 1 while Amount > 0, else 0 (deactivation flag).
  • Blocked — currently always false; sourced from stock.lot.x_kebony_blocked once that field exists.

Polling cadence: 1–5 min depending on MES throughput sensitivity.

GET /api/v1/mes/dryer-loads

Dryer Loads with nested autoclave batches and process packs.

Query params:

  • state — draft / planned / locked / done (default = planned + locked)
  • dryer — DRY1 / DRY2 / DRY3
  • ref — exact DryerLoadRef lookup

Response shape:

{
  "count": 3,
  "dryer_loads": [
    {
      "DryerLoadRef": "DL-2026-0042",
      "Id": 12,
      "Active": true,
      "Company": "Kebony BNV",
      "CreateDate": "2026-04-25T14:02:11",
      "PlanningFamily": "KRRS1",
      "Dryer": "DRY1",
      "State": "planned",
      "EstimatedStart": "2026-04-29T06:00:00",
      "EstimatedFinish": "2026-05-01T18:00:00",
      "EstimatedDurationHours": 60.0,
      "TotalM3": 24.5,
      "Packs": [
        {
          "PackNumber": 1, "PackRef": "SA-00123", "Stage": "sa",
          "ArticleNum": "RAD-95x32-3000", "InputArticleNum": "WW-RAD-95x32-3000",
          "AutoclaveBatch": 1, "PositionCol": 1, "PositionRow": 1,
          "IsHalfPack": false, "IsDummy": false,
          "PlannedLM": 567.0, "PlannedM3": 1.9051, "PlannedBoards": 162
        }
      ]
    }
  ]
}
  • Packs array is flat across both autoclave batches. Use AutoclaveBatch to regroup if needed.
  • Stage evolves over time (sa → aa → da → fa); same physical pack stays in the same JSON node via PackNumber.
  • IsDummy = true packs are filler — MES ignores for production but loads in autoclave.

Polling cadence: 1–3 min.

POST /api/v1/mes/produced-package

Called once per operation step per pack. Optional Quality + actuals fields update the parent process pack in the same request — typically only on Op 30.

Request body (Op 10 example, just the event):

{
  "OperationCode": 10,
  "ProducedPackageIdent": "MES-SA-00123-1",
  "PackRef": "SA-00123",
  "Timestamp": "2026-04-29T08:15:00Z",
  "MesRef": "WO-44912"
}

Op 30 example with Quality + actuals:

{
  "OperationCode": 30,
  "ProducedPackageIdent": "MES-FA-00123-4",
  "PackRef": "FA-00123",
  "Quality": 1,
  "ActualLM": 565.5,
  "ActualM3": 1.90,
  "ActualBoardCount": 161,
  "OutputPackQtyActual": 0.99,
  "Timestamp": "2026-05-01T18:42:00Z"
}

Required: OperationCode, ProducedPackageIdent, PackRef. All others optional.

Response:

{
  "status": "ok",
  "event_id": 17234,
  "pack_ref": "FA-00123",
  "event_type": "destacked",
  "updated_pack_fields": ["mes_quality", "actual_lm", "actual_m3", "actual_board_count", "output_pack_qty_actual"]
}
Idempotency: the API does not deduplicate. Retries create duplicate events. Middleware tracks delivery state by ProducedPackageIdent and only re-POSTs on confirmed failures (5xx / network).

POST /api/v1/mes/dryer-load/actuals

Typically once per dryer load, after the cycle ends. Updates the kebony.dryer.load record's actual start / finish / m³.

Request body:

{
  "DryerLoadRef": "DL-2026-0042",
  "ActualStart": "2026-04-29T06:00:00Z",
  "ActualFinish": "2026-05-01T18:30:00Z",
  "ActualM3": 27.85
}

Required: DryerLoadRef. All others optional.

Response:

{
  "status": "ok",
  "dryer_load": "DL-2026-0042",
  "updated_fields": ["actual_start_date", "actual_finish_date", "actual_m3"]
}

This endpoint is idempotent on the ERP side (write, not append) — safe to retry.

6. End-to-end happy path

ERP creates DL-2026-0042 (state=planned, 5 packs in AC1, 5 in AC2) │ ▼ middleware polls /dryer-loads MES picks up DL-2026-0042, drives the line │ ▼ Op 10 — Stacking begins POST /produced-package {OperationCode: 10, PackRef: "SA-00123", ...} │ ▼ Op 23 — Charged into autoclave POST /produced-package {OperationCode: 23, PackRef: "AA-00123", ...} │ ▼ Op 26 — Drying complete POST /produced-package {OperationCode: 26, PackRef: "DA-00123", ...} │ ▼ Op 30 — De-stacking with Quality + actuals POST /produced-package {OperationCode: 30, PackRef: "FA-00123", Quality: 1, ActualLM: 565.5, ActualM3: 1.9, ...} │ ▼ Cycle ends POST /dryer-load/actuals {DryerLoadRef: "DL-2026-0042", ActualStart, ActualFinish, ActualM3}
Stage prefix in PackRef. MES can either send the operation-appropriate prefix (SA- / AA- / DA- / FA-) or always send SA- and let ERP resolve. Both work since the numeric portion is stable. VVF recommends operation-appropriate for clarity in event audit. Confirm with VVF before implementation.

7. Operational guidance

7.1 Retry policy

StatusRetry?Strategy
5xxYesExponential backoff: 1s · 2s · 5s · 10s · 30s, then alert
4xxNoDeterministic failure. Log, alert, manual triage
Network / timeoutYesSame backoff as 5xx

7.2 Logging requirements

For every call (in & out), middleware logs:

  • Direction, endpoint, HTTP status, latency
  • Correlation ID (your own — e.g. MesRef)
  • For POSTs: payload (redact API key)
  • For GETs: count returned (not full body)

Retain ≥ 90 days for production troubleshooting.

7.3 Idempotency strategy

  • produced-package is not deduplicated server-side. Persist ProducedPackageIdent + ERP event_id in middleware state on success. On retry, check first; only re-POST if not previously confirmed.
  • dryer-load/actuals is idempotent (write, not append) — safe to retry.

7.4 Backpressure

ERP can absorb roughly N requests/sec/instance (TBD load-tested). For burst protection: cap concurrent in-flight requests (recommended 4–8). Queue and drain rather than parallelising aggressively.

7.5 Time skew

Both systems run NTP. Skew > 60 s causes Timestamp ordering anomalies in event histories.

8. Quick-start with curl

export BASE="https://kebonyprod-test-31051334.dev.odoo.com"
export KEY="<your-api-key>"

# Health
curl "$BASE/api/v1/mes/ping"

# Articles (BNV only)
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/articles?planning_family=KRRS1" | jq .count

# Raw packages
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/raw-packages" | jq '.packages | length'

# Planned dryer loads
curl -H "X-API-Key: $KEY" "$BASE/api/v1/mes/dryer-loads?state=planned"

# Report stacking event
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"OperationCode":10,"ProducedPackageIdent":"MES-SA-00123-1","PackRef":"SA-00123","Timestamp":"2026-04-29T08:15:00Z"}' \
  "$BASE/api/v1/mes/produced-package"

# Report DL actuals
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"DryerLoadRef":"DL-2026-0042","ActualStart":"2026-04-29T06:00:00Z","ActualFinish":"2026-05-01T18:30:00Z","ActualM3":27.85}' \
  "$BASE/api/v1/mes/dryer-load/actuals"
