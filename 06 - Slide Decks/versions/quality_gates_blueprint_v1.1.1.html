1. Design principle — stick to Odoo standard

Quality gates are control points on existing Odoo objects (receipts, pickings, manufacturing orders, deliveries, tickets) — not a parallel shadow system. Every gate:

  • Hooks into a native Odoo record via a Quality Control Point
  • Produces a Quality Check at the right moment in the flow
  • Can escalate to a Quality Alert (nonconformance) with project-management follow-up
  • Writes evidence (measurements, documents, photos) against the underlying object — not a side database
Minimal custom code. Native Odoo stages, kanbans, escalation. Custom is limited to: shared reason catalogue, Quality-Alert → Project bridge, chemical correction loop, and the €5k claim co-sign rule. Everything else is out of the box.

Process Pack model. SA / AA / DA are stages of the same kebony.process.pack record — one stable pack_number, only the stage letter changes. Display name = {stage}-{pack_number}. The pack number is the same from SA through DA. FA leaves the Process Pack model — it is a stock.lot on the finished good (1 DA → N FA split per SKU at destacking).

Container-level grouping (not pack-level merge). The "2-to-1" at the dryer is a grouping on kebony.dryer.load: 2 autoclave loads (kebony.autoclave.batch) are routed to 1 dryer run. Individual packs never merge — they carry autoclave_batch_id + dryer_load_id foreign keys to the containers they flowed through.

LevelModelScaleOwn number?
Individual packkebony.process.pack1 pile, stable from SA → DAYes — pack_number (e.g. 9999)
Autoclave loadkebony.autoclave.batchN packs (SA → AA together)Yes — its own sequence
Dryer runkebony.dryer.load~2 autoclave batches (~2N packs; AA → DA)Yes — its own sequence

Event log. Every stage transition (stacked, autoclave_in/out, dryer_in/out, destacked) is one row in kebony.process.pack.event — timestamp, operator, equipment (autoclave #, dryer #). Enables residence-time, equipment-utilisation, and full audit trail.

White wood is not a Process Pack. It stays on the supplier's own license plate. There is no "RA" internal label. "Charge" is retained as the industry term for an autoclave load — interchangeable with AA in conversation.

Note: FA as a stage label is distinct from the product code Furfuryl Alcohol (CHEM-FA) — different domain, no model collision.

Planning direction — backward from FA

Planning runs backward from the finished pack, using scrap yield:

  1. Start with FG demand (boards per SKU)
  2. Compute target SA board count: SA = ceil(FA × scrap_yield_factor) — always rounded up
  3. Planner requests white-wood plates closest to but ≥ the target
  4. Stacking machine stops at the exact board count; excess boards return to the raw-material warehouse

SA is therefore always equal to or higher than the strict yield-derived minimum — the physical constraint is supplier plate granularity.

5. Supporting models

5.1 Shared reason catalogue — quality.reason

Controlled list maintained by QA, used on every check, alert and claim. Fields: name, category (Transport / Chemical / Wood / Process / Packaging / Logistics / Other), gate_applicability (many2many — some reasons are multi-gate like transport damage), owner_user_id (default routing on claims).

Every check / alert / ticket carries reason_id (Many2one to quality.reason) + note (Text free-text).

5.4 Escalation on Quality Alert

Standard quality.alert stages, QA configures:

New → In Progress → Containment → Root Cause → CAPA → Closed, with an optional Escalated stage (auto-assign to QA Manager) triggered after repeated re-test failures. No numeric counter in code — stage-driven.

6. Chemical correction loop (Gate 6)

  Ready-Mix MO starts
        │
        ▼
  Batch produced
        │
        ▼
  Lab measurement   ◄──┐
        │              │
        In spec?        │
        │              │
  Yes ► Release · sign-off Prod + Lab · MO done
        │              │
  No  ► Adjust composition (add / dilute)
                          │
                          └──────┘  loop

Consumption accounting — lump-sum at batch close. A single consumption event records total material consumed (initial + all corrections). FIFO cost of the final mix reflects everything consumed. No per-correction JE overhead.

Autoclave block — while a Ready-Mix batch has an open alert, autoclave MOs requesting that tank are blocked (Python constraint on action_confirm).

Escalation — no numeric hard limit. After ~2–3 failed re-tests, alert moves to Escalated stage (auto-assigned to QA Manager) who decides: continue, scrap tank (exceptional), or rework with different recipe.

7. Project management bridge

Standard Odoo Project app — no parallel PM.

  • project.project + project.task with standard stages (To Do / In Progress / Done / Cancelled — QA configures)
  • Native Kanban, Gantt, deadlines, sub-tasks, followers, activities, mentions
  • Dashboards filtered per role

Native bridges:

  • helpdesk.ticket → project.project — already native (Enterprise)

Custom bridge needed (small):

  • quality.alert → project.project — one Many2one field on the alert + a "Create Project" button
  • Button spawns a project pre-filled with alert context and 3–5 template sub-tasks (Investigate / Contain / CAPA / Verify / Close)

Primary consumers: QA Manager (all quality + HSE tasks), Plant Manager (ops-tagged filter), ExCo (pre-filtered card in existing ExCo deck).

9.4 Alerting

  • Every failed QCP → Odoo activity on the owner (native)
  • Quality alert in "Escalated" stage → email to QA Manager + project task
  • Supplier tolerance breach (§10) → auto-create helpdesk ticket (type = Supplier Claim), routed to Procurement Manager
  • Gate 4 retroactive full-charge quarantine with already-shipped product → high-priority email to Sales Manager + ExCo (no auto-recall)

10. Supplier quality performance & contract tolerance

Recurring measurement of each supplier's delivered quality against their contractual tolerance, with automatic claim generation on breach. Turns per-delivery quality data into an actionable supplier-management cycle.

10.1 Model — supplier.quality.tolerance

Per supplier, per product (or category), per metric.

FieldTypeNotes
partner_idMany2one res.partnerSupplier
product_id / product_category_idMany2oneScope: specific product or whole category
metricSelectionMoisture out-of-spec % · Dimensional defect % · Cert-match failure % · On-time delivery % · Other
tolerance_valueFloatContractual threshold (e.g. 2% max moisture deviation)
unitSelectionpercent / count / days / m³
periodSelectionmonthly / quarterly
contract_referenceCharPO terms / master contract ref
valid_from / valid_toDateEffective dates
claim_template_idMany2one helpdesk.ticket.templateTemplate used if breach triggers auto-claim

10.2 Monthly computation (cron)

  1. Aggregate quality checks / alerts / receipts over the period per supplier × metric
  2. Compute actual value (e.g. % of lots failing moisture check)
  3. Compare to tolerance_value
  4. Write a supplier.quality.performance record with period, actual, tolerance, variance, breach flag
  5. If breach → auto-create a Supplier Claim ticket routed to Procurement Manager, pre-filled with: supplier + metric + period · actual vs tolerance · failing lots / PO lines · suggested claim amount (affected lot value × breach %)

10.3 Supplier performance report

Monthly PDF + live dashboard. Per supplier:

  • Volume received (m³, €)
  • Gate failures (count, %, €)
  • Breakdown per reason
  • Actual vs tolerance per metric — colour-coded green amber red
  • 12-month trend
  • Active + closed claims
  • Suggested action (warn / formal claim / supplier review)

12. Decisions made / open questions

✅ Decided (April 2026 workshop)
  • Stick to Odoo standard — minimal custom code
  • Shared quality.reason catalogue maintained by QA — controlled list + free text
  • One quarantine location per site; pack-label chain (white wood → SA → AA → DA → FA) carries identity
  • Default quarantine → defect; QA Manager sign-off required only on lab-triggered full-charge quarantine
  • Gate 3: SA always formed with good boards; bad boards → waste sorted per species
  • Gate 4: async block-by-exception; no auto-recall on shipped product, high-priority alert to Sales + ExCo
  • Gate 5 RDK: one quality report per PO; A / B / C grades as three distinct products
  • Gate 6 chemical: lump-sum consumption at close; Production + Lab joint sign-off; autoclave blocked until fixed
  • Gate 7: mandatory photo per pack; auto-draft credit note
  • Gate 8: customer portal + web form + email-to-ticket → Helpdesk; €5k co-sign threshold
  • Project: standard Odoo Project, standard stages; Alert → Project bridge as the only custom piece
  • Reason → owner mapping: build with QA Manager as project deliverable
⚠ Open
  • Furchem — supplier? lab? test method? And is Gate 1 lab measurement one test per delivery, or per chemical received?
  • MES error-code list — Word doc to come. Drives how much Gate 3 defect reason is coded vs free-text
  • Gate 6 escalation threshold — ~2–3 failed re-tests suggested; confirm with QA Manager at kick-off
  • Gate 3 sampling norm X/100 — define exact X with QA Manager
  • Gate 5 RDK legal claim mechanism on A-grade variance — shape with Procurement / Legal
  • HSE scope — Quality Alert category carries HSE as a value; confirm ownership of HSE alerts (likely same QA Manager at this stage)
  • Supplier tolerance matrix — exact metrics and thresholds per supplier / category · to be populated jointly with Procurement + QA Manager from existing contracts
  • Material breach escalation threshold (§10.5) — amount above which a supplier claim escalates to COO + Finance
