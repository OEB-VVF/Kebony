1. One Version of the Truth

The foundational principle: every number in every report traces back to one authoritative source

The Problem Today
  • Multiple spreadsheets with conflicting numbers
  • Manual reports rebuilt monthly from raw exports
  • Different people calculate the same KPI differently
  • No clear ownership of data quality
  • Cutoff issues: “is this invoice included or not?”
  • Shadow reporting: Excel files that nobody else can read
The Principle
  • One source — Odoo is the system of record for all business data
  • One extraction path — automated, scheduled, auditable
  • One BI layer — all reports from the same data set
  • One definition — each KPI has exactly one formula, one steward
The Flow (Microsoft Fabric)
SOURCE PREP & INGEST LAKEHOUSE VISUALISE Odoo 19 → Bronze (raw) Prediktor → Fabric Data ↓ Salesforce → Factory → Silver (validated) → Power BI D365 / Perfion→ ↓ AX2012 → Gold (enriched) WordPress →

The Cardinal Rule

If a number is wrong, fix it at the source (Odoo or Prediktor) — never in the report, never in the ETL, never in an Excel override. The BI layer is read-only truth.

No business logic in the BI layer. KPI definitions live in the ETL or in Odoo itself. Dashboard formulas only do presentation (formatting, filtering, drill-down). This prevents “magic formulas” that only one person understands.

2. Data Stewardship — Who Owns the Numbers

A Data Steward is accountable for a KPI’s definition, data quality, and cutoff rules. It is a business role, not an IT role.

The Four Responsibilities
Responsibility What It Means Example
Definition What does the KPI measure? What is the formula? What is included / excluded? “Net Revenue = posted invoices − credit notes, in entity currency”
Source Where does the raw data come from? Which Odoo model, which Prediktor tag? “account.move where type = out_invoice, state = posted”
Quality Is the data complete, timely, accurate? Are there gaps or duplicates? “Check for unposted draft invoices at M+5”
Cutoff When is data final? What is the closing calendar? What can change after close? “Revenue final at M+5 working days. COGS final after GRNI cleared.”
Cutoff is the #1 cause of conflicting numbers. Two people pulling the same report on different dates get different results. The steward defines when a KPI becomes official — before that, it’s a preliminary estimate.
Who Is the Steward?
KPI Family Steward Why Them
Commercial Sales Director Owns revenue targets, customer relationships, pricing
Margin CFO Owns P&L structure, COGS methodology, accrual rates
Production Operations Director Owns output targets, yield, OEE, process parameters
Supply Chain Logistics Manager Owns OTIF, lead times, inventory policy
Working Capital CFO Owns cash cycle, AR/AP policy, inventory valuation
People HR / Ops Director Owns headcount, productivity targets

The Steward Is NOT the Data Entry Person

The steward doesn’t enter invoices or close MOs. They are accountable for the KPI being correctly defined, correctly sourced, and correctly reported. If a KPI is wrong, the steward investigates — they don’t blame IT.

4. Data Architecture — Microsoft Fabric Lakehouse

Platform: Microsoft Azure · Governance: Microsoft Purview · AI/ML: Microsoft Copilot

Source: Kebony IT — Data Warehouse Architecture (Microsoft Fabric)

Bronze Layer

Raw extraction. Exact copy of source data. Immutable, timestamped. No transformation. “What the system said at that moment.”

Silver Layer

Validated & joined. Cleaned, typed, deduplicated. Odoo + Prediktor joined by lot/MO. Currency converted. This is where data quality rules run.

Gold Layer

Enriched & KPI-ready. Business formulas applied. Aggregations pre-computed. Ready for Power BI consumption. All KPI logic lives here.

Power BI

Read-only dashboards. Semantic model in Azure Data Model Library. Reports & dashboards in Report Library. No formulas — just presentation.

Governance: Microsoft Purview — data catalog, lineage tracking, access policies. Every KPI traces from dashboard → Gold → Silver → Bronze → source system. Full auditability.
AI/ML: Microsoft Copilot — natural language queries on the Gold layer. CEO asks “show me margin by country for KRRS last quarter” and gets an answer. Phase 3 capability.

5. AI-Powered BI — The Next Layer

Microsoft Copilot on the Gold layer · Natural language access to KPIs · From dashboards to conversations

The Shift: Dashboards → Conversations
Traditional BI: build a dashboard, train people to use filters and drill-down. If the question isn’t on the dashboard, request a new report. Weeks of wait.

AI BI: ask a question in plain language. Get the answer instantly from the same Gold layer that feeds the dashboards. Same data, same definitions, same truth — different interface.
Example Queries (Copilot on Gold Layer)
CEO Question What Copilot Does
“Show me GM I by product group for Q3” Queries Gold layer, returns pivot table + chart
“Which country had the highest margin drop vs last quarter?” Computes delta, ranks, highlights top mover
“What’s our cash conversion cycle trend?” Pulls DSO + DIO − DPO over 12 months, plots trend
“Why is dryer utilisation down this month?” Joins Prediktor + Odoo data, identifies root cause (maintenance, batch size, etc.)
“Compare KRRS margin this quarter vs budget” Pulls actual from Gold, budget from cost version, shows variance
Why This Works (and Why It’s Safe)

Same Source of Truth

Copilot queries the Gold layer — the exact same data that feeds Power BI dashboards. One version of the truth, whether you read a dashboard or ask a question.

Same Definitions

KPI formulas live in the Gold layer, not in Copilot’s prompts. “GM I” means the same thing whether computed by the ETL or retrieved by AI. No hallucinated formulas.

Governed by Purview

Microsoft Purview controls who can access what data. Copilot inherits the same access policies as Power BI. No data leakage, full audit trail.

The Architecture Enables This

Bronze → Silver → Gold is not just a data pipeline pattern — it’s the foundation for AI BI. Because all business logic lives in the Gold layer (not scattered across dashboards), Copilot has a clean, well-defined dataset to query. Without the Lakehouse, AI BI doesn’t work reliably.

Phase 3 capability. Requires Gold layer to be mature and validated. Stewards must have confirmed KPI definitions are correct before Copilot serves them to the CEO.

5. Data Steward × Maintenance — Keeping It Alive

KPIs degrade over time if nobody maintains them. The steward’s monthly review is the defence.

Monthly KPI Health Check
# Check What the Steward Verifies
1 Completeness All source data loaded? No missing days / batches?
2 Cutoff All invoices posted? All MOs closed? Accruals booked?
3 Definition Does the KPI still measure what it should? Any scope change?
4 Reconciliation Does the BI number match the source? Cross-check vs Odoo pivot.
5 Outliers Any suspicious spikes or drops? Investigate before publishing.
If a KPI drifts: The steward escalates to IT / data team for a source fix. Never patch data in the BI layer. Never “adjust” in Excel. Fix the root cause in Odoo or Prediktor, then the report self-corrects.

The Maintenance Calendar

M+5 WD: Financial KPIs finalised (revenue, margin, OWCR)
M+3 WD: Production KPIs finalised (output, yield, OEE)
M+10 WD: Full KPI package published to CEO dashboard

Already Built — Sales & Cash Dashboard

Sales Pipeline View

Custom SQL pivot: Invoiced + To Invoice by Sales Area × Month. Drill-down to customer, then to invoice. Volume in m³ alongside amount. Answers: “how much did each area sell, and what’s in the pipe?”

Cash Position View

Same pivot, cash filter: Paid + To Cash In + Prepayments. Answers: “how much cash came in, how much is outstanding, what prepayments are on the books?”

Margin Analysis — Already in Odoo

Every invoice carries computed fields: COGS (FIFO lot cost or dropship vendor bill), accruals (marketing, royalties, mgmt fees, rep commission), freight, warehousing. Margin = Revenue − all cost layers. Pivot by any dimension: SKU, customer, country, product group, sales area.

Drill-Down Capability

Every pivot row can be expanded: Sales Area → Customer → Invoice. Group by month, quarter, or sales rep. Export to spreadsheet for ad-hoc analysis.

The more KPIs in Odoo, the better. Every KPI computed inside the ERP is available in real-time, tagged with the 5 analytic plans, and automatically flows into the Lakehouse when Fabric goes live. No rework, no retroactive tagging, no manual export.

Odoo = Immediate. Fabric = Polished.

The CEO gets 80% of the KPIs on day 1 via Odoo native reports. Fabric adds the remaining 20% (Prediktor integration, cross-source joins, AI queries) and wraps everything in Power BI presentation. But the data foundation and business logic are already in Odoo.

14. Implementation Phasing

From Odoo go-live to full CEO dashboard · Three phases, all delivered by end of 2026

Roadmap
Phase What When Priority Depends On
Go-live Odoo configured with 5 analytic plans. All raw data available in the ERP. Stewards appointed. July 2026 Must ERP migration
Phase 2 Microsoft Fabric Lakehouse live. Bronze/Silver/Gold layers populated (nightly). Basic Power BI dashboards: Commercial + Margin + OWCR. Sep 2026 Must Fabric infra, ETL design
Phase 3 Prediktor integration. OEE + Production dashboards. Full KPI package with drill-down. Option B+ smoothed margin view. Q4 2026 Must Prediktor API, Gold layer maturity
Interim (July–Sep 2026): Between go-live and Phase 2, KPIs can be extracted directly from Odoo’s native pivot tables and exported to Excel. Not ideal, but sufficient. The analytical plans are in place — the data is tagged correctly from day 1. No retroactive tagging needed.
Critical path: The 5 analytic plans must be seeded at go-live. If data isn’t tagged from day 1, it’s very expensive to backfill. The BI platform can wait — the data structure cannot.
