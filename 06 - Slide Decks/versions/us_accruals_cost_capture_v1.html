Marketing Accrual — Detailed Flow

Obligation to reimburse distributor marketing spend up to a ceiling
What it is
  • Kebony has a contractual obligation to reimburse distributors for marketing expenses
  • The ceiling is computed as a % of sales to that distributor
  • 4% on wood products, 1.5% on accessories
  • Only applies to customers classified as "Distributor"
How the system computes it
  1. Customer invoice is posted
  2. Each product line: check if customer = Distributor
  3. If yes → apply rate × price_subtotal
  4. Store amount on x_studio_marketing_accrual_line_item
  5. Sum flows into invoice-level x_studio_total_accruals_cost
Monthly JE (EOM Accruals button)
DR 581000 – Marketing Expense $4,000 analytic: N. Rockies (sales area) CR 215510 – Marketing Liability $4,000 partner: Rex Lumber ← fund tracking!
Check Rex's fund balance
Partner Ledger → Account 215510 → Partner = Rex Lumber → Shows exactly how much Rex has left to claim
When vendor bill arrives
Rex sends invoice for $3,200 marketing spend DR 215510 – Marketing Liability $3,200 partner: Rex Lumber CR AP – Accounts Payable $3,200 → Zero P&L impact (expense was already booked) → Remaining $800 stays in Rex's fund
Important: The vendor bill MUST be booked against 215510 (liability) with partner = Rex. NOT against 581000 (expense). Otherwise you double-count.

Invoice-Level Overrides

Three toggle switches to disable accruals for specific invoices
Toggle Effect when unchecked Use case
Compute Marketing Marketing accrual = 0 for all lines Promotional shipment, no marketing obligation
Compute Royalties Royalties = 0 for all lines Internal transfer, exceptional deal
Compute Mgmt Fees Management fees = 0 for all lines Exceptional deal structure

All three default to checked (accruals active). Uncheck before clicking the margin recompute button.

Rate overrides (new)

Each rate can be adjusted per invoice in Other Info → Rate Overrides:

Marketing (Wood) default 4%
Marketing (Accessory) default 1.5%
Royalty default 5%
Mgmt Fees (Wood) default 14%

Recompute Flow

1. Open posted customer invoice 2. Adjust toggles if needed 3. Click "Recompute Margin" button 4. End-of-Month Accrual Journal Entry

The EOM Accruals button on the Accounting Hub
What the button does
  1. Scans all posted invoices with un-accrued lines
  2. Groups amounts by type × sales area × partner
  3. Creates one journal entry with paired lines:
    • DR: Expense (581000/588000/584000/586000/591000) + sales area analytic
    • CR: Liability (215510/520/530/540/560) + partner (distributor or rep)
  4. Collects freight from invoice header (manual amount)
  5. Marks all processed lines + freight invoices as accrued
Idempotent: Running the button twice won't double-accrue. Lines are flagged after first run.

Vendor Bill Clearing — Zero P&L Impact

When the actual bill arrives, it clears the liability — NOT the expense
Critical: Correct booking
Vendor Bill From Book Against With Partner
Distributor (marketing) DR 215510 / CR AP The distributor
Sales representative DR 215520 / CR AP The sales rep
Kebony Holding (royalties) DR 215540 / CR AP Kebony Holding
Kebony Holding (mgmt fees) DR 215530 / CR AP Kebony Holding
Carrier (freight) DR 215560 / CR AP The customer
NEVER book vendor bills against expense accounts (581000, 588000, etc.). The expense was already recognized via the EOM accrual. Booking against expense = double-counting.
Monthly reconciliation
  • Balance 215510: Accrued marketing minus vendor bills received = outstanding obligation
  • Over-accrual: If vendor bills < accrued → release excess at year-end
  • Under-accrual: If vendor bills > accrued → book additional expense
Partner = the key to fund tracking

Always set the Partner on liability lines

When booking the vendor bill against a liability account, always set the same partner that was used on the original accrual. This is how you track each distributor's fund balance.

To check a fund balance: Partner Ledger → filter account (e.g. 215510) + partner (e.g. Rex Lumber). The balance = amount Rex can still claim.

Freight & Customs — Closing the Landed Cost Loop

IAS 2 / US GAAP requires full acquisition cost in inventory. Today it's manual and error-prone. We automate it.
The problem today

When goods arrive from abroad, freight and customs bills often lag by 2-6 weeks. Until those costs are allocated, inventory is undervalued and margin is overstated.

The manual process requires the controller to:

  1. Remember that a receipt needs a landed cost provision
  2. Estimate freight + customs amounts
  3. Create an accrual journal entry
  4. Create a Landed Cost object in Odoo
  5. Link it to the correct receipt
  6. Choose the right split method
  7. Validate to update valuation layers
  8. When the bill arrives weeks later — reverse, redo at actual
8 manual steps, any of which can be forgotten. If step 1 is missed, inventory is wrong until someone notices — often at year-end audit.
The solution: Linked Cost POs (custom dev)

A goods PO (e.g. wood from Brazil) gets linked Cost POs for freight and customs. The system handles everything automatically:

GOODS PO ──── linked to ───┬── PO-Freight (forwarder) wood from Brazil └── PO-Customs (broker) │ ▼ receipt validated │ For each linked Cost PO: │ Bill posted? ─── YES ──→ auto-create Landed Cost │ source = vendor bill NO split across receipt lines ✓ │ ▼ Auto-create accrual JE (estimated from PO) Auto-create Landed Cost source = accrual JE split across receipt lines ✓ │ ▼ bill arrives later │ Auto-reverse accrual Auto-create new Landed Cost at actual Delta absorbed + re-split ✓
Zero manual steps. The controller only needs to create the Cost POs at purchase time. Everything else — accrual, allocation, settlement — is automated.

Automated Landed Cost — How It Works

Three triggers, fully automated, using Odoo's native Landed Cost engine for proper cost allocation
Trigger 1 — Receipt Validated
Receipt WH/IN/00123 validated System finds goods PO → finds linked Cost POs: PO-Freight ($8,500) — no bill yet → Auto-create accrual JE: DR 110410 Landed Cost Clearing $8,500 CR 211000 GRNI Accrual $8,500 → Auto-create Landed Cost: Transfer: WH/IN/00123 Source: accrual JE Split: by value across receipt lines → Validate → valuation layers updated PO-Customs ($2,200) — bill already posted → Auto-create Landed Cost: Transfer: WH/IN/00123 Source: vendor bill Split: by value across receipt lines → Validate → valuation layers updated Chatter message on goods PO: summary of what was created
Trigger 2 — Vendor Bill Posted
Freight bill arrives: $9,100 (vs $8,500 estimated) System detects: this bill's PO is a linked Cost PO → Auto-reverse accrual JE ($8,500) → Cancel estimated Landed Cost → Auto-create new Landed Cost at actual ($9,100) Source: vendor bill Split: by value across original receipt lines → Validate → valuation layers corrected → Delta: +$600 properly split across products
Trigger 3 — EOM Dashboard

Pending Landed Cost Accruals

New section on the Accounting Hub showing all outstanding estimated Landed Costs:

Receipt Goods PO Cost PO Est. Amount Days Open
WH/IN/00123 PO-001 PO-Freight $8,500 42 days
WH/IN/00156 PO-012 PO-Customs $1,800 12 days

Items > 30 days highlighted as overdue. Controller follows up with vendor.

Why native Landed Cost matters

Proper cost allocation vs lump-sum journal entry

A manual JE (DR Inventory / CR Accrual) puts a lump sum on inventory without distributing it across products on the receipt.

By using Odoo's native Landed Cost object, the system splits the cost across receipt lines using the chosen method (by value, quantity, or weight). Same quality of allocation whether the bill has arrived or not.

Analytical Tags — Sales Area

Every transaction must carry the right Sales Area for reporting
Where Sales Area lives
Document Field Populated by
Customer (partner) x_studio_area_of_sales Master data (manual)
Customer Invoice x_studio_area_of_sales Auto from customer
Invoice line analytic_distribution Auto from invoice/SO
EOM Accrual JE analytic_distribution Copied from invoice lines
Vendor bill (clearing) analytic_distribution MANUAL — must match
Critical: When booking a vendor bill against a liability account (2155xx), the accounting team must set the same Sales Area analytical tag as the original accrual. Otherwise the liability balance by area becomes unreliable.
Why it matters

Profitability by Sales Area

Management needs margin analysis by geographic area. If revenue is tagged "East Canada" but the corresponding accrual clearing is untagged (or tagged differently), the area P&L becomes meaningless.

Reconciliation by Area

Monthly liability reconciliation (215510/520/530/540) should be done per Sales Area. Each area should show:

  • Opening balance (accrued)
  • + New accruals (EOM JE)
  • − Vendor bills received
  • = Closing balance (outstanding obligation)

Future: Product Group Analytic

A second analytical dimension (KSP, KRRS, KR) will be added for product-group profitability. Same tagging discipline applies.

Initial Backfill — Jan 2026 to Today

One-time catch-up to accrue all historical invoices since January
Why this is needed

Invoices posted since January 2026 have accrual amounts computed on each line (via margin button) but no journal entry has been posted yet. The liability accounts (2155xx) are empty.

We need a catch-up JE to recognize all accruals from Jan 1 to today.

3-Step Procedure
# Action Where
1 Recompute Margin on all historical invoices
(ensures accrual amounts are populated on lines) 		Each invoice → "Recompute Margin (US)"
2 Reset Accrual Flags
(clears is_accrued so EOM picks up all lines) 		Accounting Hub → "Reset Accrual Flags"
3 Run EOM Accruals
(creates one catch-up JE for all historical invoices) 		Accounting Hub → "Run EoM Accruals (US)"
What the catch-up JE looks like
Catch-Up Accrual JE — March 2026 Covers: Jan 1 → Mar 31 2026 (all invoices) ──────────────────────────────────────────────── DR 581000 Marketing $48,200 grouped by sales area CR 215510 Mkt Liability $48,200 per distributor (partner) ──────────────────────────────────────────────── DR 588000 Sales Rep $12,800 CR 215520 Rep Liability $12,800 per sales rep (partner) ──────────────────────────────────────────────── ... royalties, mgmt fees, freight ...
Date: The catch-up JE is dated today. If you need monthly splits for cleaner period reporting, reverse this JE and re-run with date filters per month (future enhancement).
Warning: "Reset Accrual Flags" is for initial setup only. In normal monthly operations, the EOM engine is incremental — it only picks up new, un-accrued invoices.
