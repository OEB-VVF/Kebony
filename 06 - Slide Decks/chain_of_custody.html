Why Chain of Custody in the ERP?

The auditor's question: "Show me the chain from this invoice back to the source."

Today's reality

  • 3 Studio fields on product + company (static text)
  • FSC code printed on BOL/invoice via QWeb template fallback
  • No lot-level traceability
  • No claim propagation through stock moves or manufacturing
  • No evidence (attachments) linked to receipts
  • No supplier certificate verification records
  • Hardcoded species/origin text on Reception BOL

What an auditor expects

  • Inbound claim evidence per lot (supplier cert + source doc)
  • Material accounting records (input % → output claim)
  • Unbroken traceability: receipt → stock → production → shipment
  • Sales document control (claim + cert code on delivery/invoice)
  • Nonconforming product records
  • Supplier certificate register
  • 5-year record retention
Gap: Today we can print FSC on documents, but we cannot prove the claim. An auditor asking "which certified lots went into this shipment?" gets no system answer.

Control methods supported
Method Description Status
Percentage Output claim = f(% certified input in period) Active today
Physical separation Certified / non-certified kept apart Future (100%)

Design principle: The system tracks certified % per lot and per period. When input reaches 100%, no process change is needed — the math just gives "FSC 100%" output automatically.
Scope: All Kebony entities globally. Both FSC and PEFC — scheme depends on country of origin and supplier. Certificate codes differ by company (INC, BNV, Holding). Multi-company + multi-scheme natively supported.

Design Principle

Certification travels with the material flow
INBOUND INTERNAL OUTBOUND ─────────────────────── ────────────────────────── ─────────────────────── Vendor doc + cert Internal transfer Delivery picking │ │ │ ▼ ▼ ▼ PO Line Move lines inherit Claim validated (planned claim) cert data from lot against source lots │ │ │ ▼ ▼ ▼ Receipt Dryer Load → MO Delivery slip prints: │ cert validation at scheme + claim + code ▼ action_done │ Lot created │ ▼ claim + supplier cert ▼ Invoice mirrors claim + evidence attached cert.trace.link │ input → output ▼ stock.move.line cert data stored
Key rule: Users never retype certification data downstream. It auto-propagates from receipt lot through every movement to the final shipment. The system, not the user, guarantees the chain.

Business Rules

Five rules that make the system audit-proof
# Rule What happens Enforcement
1 Receipt gating If product is cert-relevant: require scheme, claim (%), supplier cert code, lot. If missing → soft warning or quarantine. Configurable per entity. Warn / Block
2 Claim propagation Cert data auto-flows: receipt lot → internal moves → MO → delivery. No manual retyping. All wood products are lot-tracked (no exceptions). Automatic
3 Mixing control Kebonisation and RDK MOs: hard block, no override — all consumed lots must share the same scheme & claim. Other MO types (repacks, grade-downs): warn or block, configurable. Override requires justification → logged as nonconformity. Hard block + Warn/Block
4 Outbound validation Before delivery with certified claim: verify source lots exist, qty supports claim, evidence attached, cert code available for printing. Warn / Block
5 Audit packet From any lot/MO/picking/invoice: open full traceability view — source docs, movements, trace links, output docs, nonconformities. One-click
Simplicity principle: Warn by default, block only where configured. Bulk receipts (containers) support partial claim with per-lot override. The system must help the warehouse, not slow it down.

Audit Reports

Seven reports that answer every auditor question
# Report What it proves Phase
1 Certified inbound register Date, supplier, cert code, product, claim in, qty, lot, source doc, attachment status MVP
2 Certified stock by lot/location Lot, product, location, scheme, claim, qty on hand, hold flag — proves segregation MVP
3 Input-to-output traceability From any shipment: inbound lots consumed, qty, resulting claim — the core audit report MVP
4 Certified sales register Delivery, invoice, customer, product, qty, claim out, cert code, linked lots MVP
5 Supplier certificate register Supplier, scheme, cert code, validity dates, verification evidence MVP
6 Claim ledger / annual volume summary By period + product group: certified input, total input, %, output claimed, closing balance MVP
7 Nonconformity / corrective action Issue type, lot, action taken, status — proves governance Phase 2
Auditor test: Start from any invoice → click through to source lots → see supplier cert → see % calculation → confirm claim is supported. Should take < 2 minutes.

April 2026 Workshop — Outcomes

Five decisions hardening the architecture

1 · MO mixing — hard block

Kebonisation and RDK MOs must not mix different certifications. Enforced in Python at action_confirm and action_done. No override. Other MO types keep the warn/block toggle.

2 · SO certification toggle

New toggle on sale.order.line mirroring the minimum-quantity-guarantee toggle. Fields: scheme required + claim required. Filters reservation + gates delivery.

3 · RDK subcontract validation

Contractual no-mixing clause + technical check. Inputs blocked if mixed. Output claim declared by subcontractor reconciled against inputs → mismatch auto-quarantines & logs a nonconformity.

4 · Invoice = proof of certification

Every certified line prints scheme + claim + cert code. For US-bound lines: add ESR number (ICC-ES). Reconstruction chain invoice → delivery → lot → vendor must hold for 5 years.

5 · New axis — ICC-ES. US building-code compliance certification added alongside FSC/PEFC. Lives on product.template, not on the lot. US market only. No mixing concept — it's a SKU attribute. See next slide for detail.

ICC-ES — Second Certification Axis

US building-code compliance · sits alongside FSC/PEFC, not inside it
Dimension FSC / PEFC ICC-ES
Purpose Chain of custody — origin & handling of the wood Building-code compliance — performance of the finished product
Carrier stock.lot (per-lot) product.template (per SKU)
Mixing concept Yes — mixing at MO contaminates the claim No — it's a SKU attribute
Market scope Global (customer-driven) US only (code-driven)
Renewal Annual CoC audit Annual fee + retained testing evidence
Surfaced on Invoice line: scheme + claim + cert code Product spec sheet; submittal PDF; invoice footer
Risk of loss Contamination / missing evidence ESR expiry / recipe change / QA gap

FieldPurpose

x_kebony_icc_esr_valid_toExpiry — triggers renewal alert 90 days out
x_kebony_icc_esr_scopeCovered use categories (structural, decking, cladding, fire…)
x_kebony_icc_attachment_idsESR PDF + underlying test reports
x_kebony_icc_us_relevantGates ICC checks to US-bound products only
Governance. 90-day expiry alert. Lapsed ESR on a US-bound SO line = hard block (regulatory risk, not commercial).
